Implementation Approach

One strong approach recommended — here's how it works. Approve or adjust before we write the spec.

Architecture: 9-Patch Tile Registry

Frame art stored as TypeScript string arrays — no file I/O, consistent with codebase patterns

┌─ TL corner (fixed) ─┐ topEdge tile (repeats→) ┌─ TR corner (fixed) ─┐ │ ~10w × ~10h chars │ ───────────────────── │ ~10w × ~10h chars │ leftEdge (repeats↓) content lines rightEdge (repeats↓) │ ~6w chars │ string[] │ ~6w chars │ └─ BL corner (fixed) ─┘ bottomEdge tile (repeats→) └─ BR corner (fixed) ─┘

Each of the 6 variants is a FrameTileSet object. The renderer composes them at any width/height by tiling edges between fixed corners.

Color System: Depth Mapping

Block density automatically maps to 4 theme color slots — no per-char configuration needed

░→shade1 — shadow / darkest
▒→shade2 — mid-shadow
▓→shade3 — mid-highlight
█→shade4 — brightest / foreground
─│╔→shade3 — box-drawing chars

Each theme defines its own shade1–shade4 defaults. Users can override any slot per-frame instance.

Usage API

const frame = new OrnateFrame({
  variant: 'baroque',
  colors: { shade4: '#ffd700' } // optional override
});

const output: string[] = frame.render(
  contentLines,  // string[]
  80,            // width
  24             // height (optional)
);
